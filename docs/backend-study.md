# 📔 Spring Boot Backend Study Note

본 문서는 스프링 부트 백엔드 애플리케이션을 개발하고 공부하는 과정을 기록한 학습 노트입니다. 각 기능 구현 단계별로 습득한 스프링 부트의 주요 동작 원리와 아키텍처 개념을 정리합니다.

---

## 🛠️ Chapter 1: 프로젝트 초기 설정 & 환경 격리 (Task 1)

### 1. YAML (`.yml`) 설정 파일이란?
* 스프링 부트는 애플리케이션 기동 시 필요한 모든 설정 정보(DB 연결 정보, 포트 번호, 외부 API 키 등)를 `src/main/resources/` 하위의 설정 파일에서 읽어옵니다.
* 과거에는 `application.properties` (Key-Value 형식)를 주로 썼지만, 최근에는 계층 구조를 직관적으로 파악할 수 있는 **YAML (`.yml`)** 형식을 널리 사용합니다.

### 2. 멀티 프로파일 (Active Profiles) 격리 원리
* **배경**: 로컬 개발 시에는 가볍고 초기화가 빠른 DB(H2)를 써야 하고, 실제 서버(운영)에서는 Supabase 같은 영구 DB를 연결해야 합니다. 소스코드를 직접 수정하지 않고 환경을 격리하기 위해 **프로파일(Profile)** 기법을 씁니다.
* **동작 방식**:
  - `application.yml`: 공통 및 기본 활성화 프로파일 지정.
  - `application-local.yml`: 로컬(local) 환경 전용 설정. `spring.profiles.active: local`로 기동할 때 활성화됩니다.
  - `application-prod.yml`: 실제 배포(production) 환경 전용 설정.
* 스프링 부트는 `application-{profile}.yml` 규칙으로 작성된 파일들을 찾아 현재 활성화된 프로파일(`spring.profiles.active`)의 설정을 우선하여 덮어씌웁니다.

### 3. H2 인메모리 데이터베이스란?
* 메모리 상(RAM)에만 존재하는 가볍고 임시적인 관계형 데이터베이스(RDB)입니다.
* 서버가 켜질 때 테이블이 만들어지고, 꺼지면 데이터가 모두 날아가기 때문에 **로컬 개발 테스트 및 신속한 기능 검증**에 최적입니다.

### 4. JPA 엔티티와 인덱스 (Index) 개념
* **JPA (Java Persistence API)**: 자바 객체와 DB 테이블을 매핑해주는 기술(ORM)입니다.
* **`@Entity`**: 이 클래스가 데이터베이스의 테이블과 1:1로 매핑되는 객체(엔티티)임을 스프링 데이터 JPA에 알려주는 어노테이션입니다.
* **`@Table(indexes = ...)`**: 
  - **인덱스(Index)**는 데이터베이스에서 책의 '찾아보기(색인)'와 같은 역할을 합니다.
  - 질문 목록을 검색할 때 `category`와 `subject` 기준으로 주로 검색한다면, 인덱스가 없을 때는 DB가 처음부터 끝까지 모든 행을 뒤져야 하지만(Full Scan, $O(N)$), **복합 인덱스(Composite Index)**를 생성해두면 매우 빠르게 데이터를 찾아낼 수 있어($O(\log N)$) 성능이 극대화됩니다.

## 📂 Chapter 2: 공통 API 규격 & 글로벌 예외 처리 (Task 2)

### 1. 공통 응답 객체 (Response Wrapper)의 필요성
* **문제**: 어떤 API는 결과값으로 `List<Question>`을 반환하고, 다른 API는 단순 문자열을 반환하거나, 에러 시 톰캣 내부 에러 스택을 그대로 리턴하면 프론트엔드 개발자는 데이터를 수신할 때마다 JSON 구조를 분석해야 하므로 연동 편의성이 크게 떨어집니다.
* **해결**: 모든 API의 응답을 통일된 규격(`ApiResponse`)으로 한 번 감싸서(Wrap) 반환합니다.
  - 성공 시: `{"success": true, "data": [실제 데이터], "error": null}`
  - 실패 시: `{"success": false, "data": null, "error": {"code": "코드", "message": "원인"}}`

### 2. `@RestControllerAdvice` 와 글로벌 예외 처리 (Exception Handling)
* **`@RestControllerAdvice`**: 스프링 부트 내부의 모든 컨트롤러(`@RestController`)에서 발생하는 예외(Exception)를 감지하고, 에러 처리를 한곳에서 중앙 집중식으로 가로채어(Intercept) 처리할 수 있도록 돕는 매우 유용한 어노테이션입니다.
* **`@ExceptionHandler`**: 특정 예외 타입(예: `IllegalArgumentException` - 잘못된 인자 입력 에러)이 감지되었을 때 작동할 구체적인 에러 핸들러 메소드를 선언하는 역할을 합니다.
* **보안 측면**: 데이터베이스 에러나 서버 로직 에러(500)가 발생했을 때 내부의 복잡한 스택 트레이스(StackTrace)를 클라이언트 브라우저에 그대로 뱉는 것은 해킹 위협 등 보안상 금기 사항입니다. 글로벌 예외 처리기를 두어 서버는 안전하게 로그(`log.error`)로 에러 이력을 기록하고, 사용자에게는 정제된 에러 메시지만 응답하도록 가공해 주는 것이 필수적입니다.

## 💾 Chapter 3: Repository 설계, Native Query & TDD (Task 3)

### 1. Spring Data JPA Repository의 이해
* 스프링 부트에서 데이터베이스 테이블에 접근하기 위해서는 인터페이스로 `JpaRepository<EntityClass, PrimaryKeyType>`를 상속받은 인터페이스를 정의합니다.
* 스프링이 실행될 때 이를 구현한 프록시 객체를 동적으로 구현하여 내부적으로 주입해주므로, 개발자는 SQL을 1줄도 작성하지 않고 `save()`, `findAll()`, `findById()` 등의 기본적인 데이터 조작 기능을 무료로 얻을 수 있습니다.

### 2. Native Query (네이티브 쿼리)의 선택 이유
* **Native Query**: 데이터베이스의 SQL 표준 및 특정 벤더(PostgreSQL, H2 등) 문법을 가리지 않고, 데이터베이스 전용 로우 SQL을 그대로 직접 작성해 실행하는 쿼리입니다.
* **사용 목적**: JPA 표준 객체 쿼리(JPQL)는 특정 데이터베이스 벤더 특화 기능인 `RANDOM()` 정렬이나 `LIMIT` 쿼리를 표현하기가 매우 번거롭거나 표준 방언에 제약을 받습니다. 따라서 질문 데이터 풀에서 고속으로 무작위 추출을 수행할 때는 데이터베이스 레벨에서 직접 연산이 가능한 네이티브 쿼리 `@Query(nativeQuery = true)`를 적용합니다.

### 3. TDD (Test-Driven Development)의 흐름
* **개념**: "테스트 주도 개발"은 실제 서비스 로직을 코딩하기 전에 **먼저 그 기능이 잘 돌아가는지 검증하는 테스트 코드를 우선 작성**하는 개발 기법입니다.
* **흐름 (Red-Green-Refactor)**:
  1. **RED (실패)**: 컴파일 에러나 테스트 실패를 유도하는 테스트 코드를 먼저 짭니다.
  2. **GREEN (성공)**: 그 테스트를 통과시킬 수 있는 가장 심플하고 최소한의 기능 코드를 본 소스코드 영역에 짭니다.
  3. **REFACTOR (리팩토링)**: 테스트 코드가 성공 상태를 유지하는 것을 믿고 안심하며 소스코드의 구조를 가독성 좋고 견고하게 정돈합니다.

## 💾 Chapter 4: 비즈니스 서비스 설계 & Mock 가상 검증 (Task 4)

### 1. 서비스 레이어 (Service Layer)의 역할
* **`@Service`**: 이 클래스가 비즈니스 도메인 로직을 처리하는 핵심 서비스 컴포넌트임을 스프링 부트에 선언하는 어노테이션입니다.
* **역할**: 컨트롤러와 데이터 레포지토리의 가교 역할을 담당합니다. 주로 클라이언트로부터 넘어온 데이터의 유효성 검증(Null, 공백 문자, 범위 초과 데이터 조정)을 처리하고, 트랜잭션(`@Transactional`) 경계를 선언하여 DB 작업의 원자성을 보장합니다.

### 2. 가상 목 객체 (Mockito)를 활용한 슬라이스 테스트
* **Mockito**: 테스트 대상인 서비스 로직에만 온전히 집중하기 위해, 서비스가 의존하고 있는 외부 리포지토리(`QuestionRepository`)를 가상의 복제품(**Mock**)으로 갈아 끼워 검증하는 프레임워크입니다.
* **장점**:
  - 실제 데이터베이스를 띄우고 지우는 I/O 비용이 들지 않아 테스트 실행 속도가 밀리초(ms) 단위로 극도로 빠릅니다.
  - 리포지토리의 반환 값을 개발자가 임의로 지정(`when(...).thenReturn(...)`)하여 다양한 예외 경계 상황(예: 파라미터가 비어있을 때, 음수 count가 들어왔을 때)을 가상으로 시뮬레이션하고 검증할 수 있습니다.

## 📂 Chapter 5: REST API 설계, CORS 보안 & MockMvc (Task 5)

### 1. CORS (Cross-Origin Resource Sharing)와 보안 설정
* **CORS**: 브라우저가 보안을 위해 다른 출처(도메인, 포트)에서 온 리소스 호출을 기본적으로 차단하는 보안 정책입니다.
* **현상**: 리액트 프론트엔드는 `http://localhost:5173` 포트에서 실행되고, 스프링 백엔드는 `http://localhost:8080` 포트에서 실행되므로 출처가 달라 통신 시 브라우저에서 차단 에러가 뜹니다.
* **해결**: 스프링의 `WebMvcConfigurer`를 상속받는 설정 클래스(`WebConfig`)를 생성하여, 우리가 허용하는 신뢰할 수 있는 도메인(`localhost:5173`)에 대해서만 API 요청을 허락하도록 화이트리스트 보안 설정을 구성해야 합니다.

### 2. `@RestController` 와 REST API 구조
* **`@RestController`**: 이 클래스가 JSON/XML 등의 정제된 텍스트 형식으로 클라이언트에게 다이렉트로 데이터를 응답하는 API 종단점(Endpoint)임을 나타내는 어노테이션입니다. (`@Controller` + `@ResponseBody`의 결합 형태)
* **`@GetMapping`**: HTTP GET 요청 메소드를 특정 자바 핸들러 메소드와 매핑시킵니다.
* **`@RequestParam`**: HTTP 쿼리 스트링 파라미터(예: `?category=Frontend&count=5`)를 자바 메소드의 인자로 파인딩해줍니다. `required = false` 옵션을 주면 해당 파라미터 누락 시에도 예외 없이 구동되도록 유연하게 작동합니다.

### 3. MockMvc를 활용한 API 연동 테스트
* **MockMvc**: 스프링 부트 애플리케이션을 실제로 톰캣 서버에 띄우지 않고, 스프링 MVC 웹 레이어의 가상 환경(Mock)을 만들어 컨트롤러의 API 응답 속도, HTTP 상태 코드(200 OK, 400 Bad Request), 응답 JSON 경로(`jsonPath`)를 고속으로 검증할 수 있도록 지원하는 핵심 테스트 프레임워크입니다.
* **`@WebMvcTest`**: 다른 무거운 DB 설정이나 비즈니스 의존 클래스는 빼고, 웹 API와 오직 지정한 컨트롤러 단독만을 대상으로 가볍게 슬라이스 테스트를 설계할 때 활성화하는 강력한 테스트 도구입니다.
